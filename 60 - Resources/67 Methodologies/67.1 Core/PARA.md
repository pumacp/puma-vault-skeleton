# PARA

> Organize information by **actionability**, not by topic: every note belongs to a Project, an Area, a Resource, or an Archive.

## How it works
PARA sorts everything you keep into four categories, ordered by how soon you will act on it:

- **Projects** — short-term efforts with a goal and an end date (vault folder `40 - Projects`).
- **Areas** — long-term responsibilities you maintain with no end date (`50 - Areas`).
- **Resources** — topics and references useful across projects (`60 - Resources`).
- **Archives** — inactive items from the other three (`70 - Archive`).

A note moves between categories as its actionability changes — an active Project becomes an Archive when finished; a Resource becomes a Project when you start using it for a goal.

## Source
Tiago Forte, *Building a Second Brain* (2022), and "The PARA Method" (Forte Labs). PARA is the **Organize** layer of the CODE workflow.

## Worked examples
- **Example 1 — A conference talk.** A half-finished talk with a delivery date is a **Project** (`40 - Projects/_EXAMPLE-Project`). The standing skill "public speaking" is an **Area** (`50 - Areas`). When the talk is delivered, the project folder is moved to **Archive** (`70 - Archive/72 Completed`).
- **Example 2 — A saved paper.** A PDF you bookmark "to read someday" is a **Resource** (`60 - Resources`). The moment you cite it inside an active goal, you link it from the relevant **Project**. When that project closes, the link stays but the project archives — the Resource remains reusable.
